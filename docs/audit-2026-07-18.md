# Digital Vargani — Full Product Audit (2026-07-18)

Codebase: `C:\Projects\vinayak-mandal` · Stack: React 18 + TS + Vite + Tailwind + PWA (Dexie offline queue) + Supabase
Method: static review by two specialist passes (security, architecture/correctness) + manual trace of the signup, volunteer, and admin money flows. Build/tests were not executed (sandbox unavailable) — run `npm run typecheck && npm run test -- run && npm run build` to confirm green before acting.

## Verdict

**Not production-ready yet, but much closer than expected.** The foundation is strong: cross-tenant RLS isolation is sound, the money core (`money.ts`/`reconcile.ts`) is pure, proven with hand-computed test values, and held to 100% coverage; the offline queue is idempotent and race-safe; receipt numbering is gapless per mandal. What blocks launch is a short list of correctness and flow gaps — including both issues you noticed, which are real bugs, not misunderstandings.

## Your two questions, answered

### 1. "Signup forces mandal creation — what if the user is a volunteer?"

Confirmed. `RequireRole.tsx:32-34` sends **any** authenticated user without a `users` row to `/signup`, and `Signup.tsx` is exclusively a "create your mandal" form. There is no "I was invited / I'm a volunteer" path anywhere in the funnel.

Ways a real volunteer hits this wall:

- They sign in with Google or a magic link directly (e.g. from the landing page) instead of tapping their invite link → no `users` row → forced into "Create your Mandal".
- Their invite was already redeemed once (tokens are one-time: `redeem_invite` nulls the token) and their anonymous session is later lost — cleared browser data, iOS storage eviction, new phone → **locked out permanently** with no recovery path; the only door shown is "create a mandal".
- `AuthProvider.tsx:8-15` swallows fetch errors into `null`, so even a real admin/volunteer on a flaky network can get bounced to the create-mandal screen (`fetchAppUser` failure ≠ "no row", but the code treats them the same).
- The invite error state (`InviteRedeem.tsx:78-86`) is a dead end: "invite invalid", no explanation, no "ask your admin to re-invite" action.

Also risky: opening any invite link **signs out whoever is currently logged in** (`InviteRedeem.tsx:47-50`). An admin who taps a volunteer's link to test it gets logged out and consumes the token.

**Fixes (in order):**
1. Turn `/signup` into a fork: "Create a new Mandal" vs "I was invited as a volunteer/admin" (the second explains: open the invite link your admin sent on WhatsApp, with a re-request hint).
2. Make invites re-issuable: admin screen gets a "resend/regenerate invite" action per volunteer; `redeem_invite` allowed to re-bind a volunteer whose session was lost (server-side, with the old auth user invalidated).
3. Distinguish "no row" from "fetch failed" in `AuthProvider` and show a retry state instead of redirecting.
4. Guard `InviteRedeem` with a confirm step when an existing session is present ("You're signed in as X — continue and switch?").
5. Long-term: replace anonymous-session volunteers with real (Google/phone-less magic-link) identities so sessions are recoverable. The anonymous model is the root cause of the lockout class. Note `config.toml` has `enable_anonymous_sign_ins = false` — if prod mirrors it, invites are broken entirely; reconcile this.

### 2. "Why does money collection on the admin side show a volunteer URL?"

Confirmed. The collection form is mounted **only** at `/volunteer` (`router.tsx:64-71`, allowed for both roles), and the admin dashboard's "Collect donation" card points straight there (`MasterLedger.tsx:17`). So an admin collecting money sees `/volunteer` in the URL, and back-navigation labels flip to volunteer-flavored copy (`Collections.tsx:79-81`). It's a route-naming problem: the route encodes a *role* where it should encode a *task*.

**Fix:** rename the shared flow to role-neutral paths — `/collect`, `/collect/pending`, `/collect/history` — and keep `/volunteer/*` as redirects for old links. Cheap, purely mechanical.

**But there's a bigger bug hiding under this one:** the reconciliation identity assumes cash is only ever collected by volunteers (`reconcile.ts:102-104`, proven in `tests/reconcile.test.ts:503-549`). The moment an admin logs a **cash** donation via that very "Collect donation" card, the books-balance indicator goes **permanently red** with an unexplained discrepancy. The dashboard actively invites the one action that breaks its own headline feature. Fix properly by extending `cashHeldByTreasurer` to include admin-collected cash (restores the algebra — touches the reconciliation identity, so re-verify tests), or as a stopgap hide the cash mode for admin sessions.

## Critical (fix before any real festival use)

1. **Admin cash donation breaks books-balance** — see above. This is the flagship feature failing on a natural action.
2. **Voids are forgeable and reversible.** The append-only trigger (`forbid_financial_edit`) deliberately skips the four void columns, and volunteers can UPDATE their own rows. Via a raw REST call a volunteer can: void their own cash donation (pocket the cash — the ledger stops expecting it and the books still balance), set `voided_by` to the admin's id with a backdated `voided_at`, then un-void later if questioned. This defeats the entire audit-trail promise. **Fix:** move voiding into a SECURITY DEFINER RPC that stamps `voided_by/voided_at` server-side; make void one-way in the trigger (reject `voided true→false` and any edit to void metadata once voided); consider admin-only voids for donations/handovers. (`20260717120000_multi_tenancy.sql:145-183, 231-281`, `src/lib/db/void.ts`)
3. **Actor FKs aren't tenant-checked → silent cross-mandal ledger corruption.** `collected_by`/`paid_by`/`volunteer_id`/`received_by` reference `users(id)` with no check that the actor belongs to the row's `mandal_id`. Combined with the device-scoped (not session-scoped) Dexie outbox, a shared phone can sync mandal A's queued donation into mandal B's books — counted in B's totals, invisible in every cash-in-hand view, permanently red indicator. **Fix:** `unique(id, mandal_id)` on `users` + composite FKs on all three financial tables, and bind outbox rows to the auth user (sync only the current user's rows; fence/clear on sign-out — also a donor-PII concern on shared devices).

## High

4. **Volunteer lockout / no volunteer entry path** — question 1 above.
5. **`redeem_invite` trusts the client.** It doesn't assert session type (its mirror `create_mandal` rejects anonymous sessions; this rejects nothing) — harden server-side. (`20260714124014_redeem_invite.sql`)
6. **Poison outbox items retry forever.** Any non-duplicate error is treated as "retry later"; a permanently rejected entry shows "waiting for signal" forever. Classify errors; after N non-network failures surface "needs attention". (`sync.ts:66-70`, `PendingSend.tsx:119-121`)
7. **No automated RLS verification.** Policies were rewritten wholesale in the multi-tenancy migration; unit tests mock Supabase and e2e stubs the network, so the product's entire security boundary is untested in CI. Wire `supabase/verify-local.sh` + pgTAP-style cross-tenant and void-integrity tests into `npm test`.

## Medium

8. **Donor phone mandatory in validation, nullable in schema** — a donor who declines to share a phone can't be logged at all. Validate only when non-empty. (`validation/donation.ts:35-38`)
9. **Unbounded list queries** — collections/expenses/handovers/ledger fetch entire (mandal-scoped) tables; will degrade on low-end Androids for big mandals. Add pagination or a range cap. (`db/donations.ts:96-103`, `db/ledger.ts:12-17`)
10. **Offline queue covers donations only** — expenses/handovers logged on bad signal hard-fail. Either extend the queue or set expectation in the UI.
11. **Docs drift is an active hazard.** SPEC.md still describes single-mandal `mandal_config`, Supabase Storage, Recharts — all replaced (mandals table, Cloudinary, hand-rolled FundDonut). `tasks/todo.md` shows 0/16 done while all 16 are implemented. Since these docs drive your agent sessions, stale spec = future bad edits. Update both.

## Low / polish

12. Route/copy leftovers of the single-mandal era: `strings.ts:3` still says "Vinayak Yuvak Mandal (VYM)" in a multi-tenant product branded Digital Vargani; delete stray `.gitkeep` files sitting next to real screens.
13. `formatINR` renders `₹10.5` and `₹-40`; pin 2 fraction digits and handle sign explicitly.
14. PWA manifest ships only an SVG icon with combined `any maskable` purpose and no apple-touch-icon → poor install icon on iOS. Add 192/512 PNGs.
15. `"test": "vitest"` runs watch mode — use `vitest run` for the pre-commit ritual.
16. `enable_anonymous_sign_ins=false` in `config.toml` vs the invite flow needing it — reconcile with prod.

## What's already good (don't rebuild)

Cross-tenant RLS is thorough — every policy carries a `mandal_id` predicate, every leaky RPC was re-scoped with reasoning documented inline; no DELETE policies anywhere (hard-delete impossible); `mandal_id` and `receipt_no` are server-stamped and race-safe; public receipt/transparency leak no PII (no donor phone, 128-bit unguessable tokens, aggregates only); the money core has an inline algebraic proof and independently hand-computed test expectations; the Cloudinary edge function derives the upload folder server-side; no secrets in the bundle; comment discipline throughout is excellent.
