## Part 2 — Answers to your questions

### "Donor opens the receipt link and gets 404 — why does it work on my phone?"

**Diagnosed.** There is **no SPA fallback rewrite** on the host: no `vercel.json`, no `public/_redirects`, nothing. `/r/<token>` is a client-side route — the server has no file there, so it returns 404. It works on *your* phone because you've used the app: the PWA service worker is installed and serves `index.html` for any navigation (workbox `navigateFallback`). The donor has never visited the site, has no service worker, and hits the raw 404.

**Fix (5 minutes):** add `vercel.json` at repo root:
```json
{ "rewrites": [{ "source": "/(.*)", "destination": "/index.html" }] }
```
(or the equivalent for whatever host you deploy to). This is the single highest-priority fix — receipts are currently broken for every donor.

### "Can the receipt send automatically from the volunteer's number without opening SMS/WhatsApp?"

**No — and it's an OS security boundary, not a missing feature.** No website can silently send an SMS or WhatsApp message from the user's own number; that would let any web page impersonate you. The options that do exist all change the sender and cost money:

- **WhatsApp Business Cloud API** — sends from a *mandal business number* (not the volunteer's), needs Meta approval, and utility template messages cost roughly ₹0.12–0.35 each in India. Loses the "message from a trusted neighbor" property your brief explicitly values.
- **SMS gateway (MSG91/Twilio/etc.)** — sender becomes a DLT-registered header, per-message cost, registration paperwork.

Your current deep-link flow (one tap → composer pre-filled → press Send) is the zero-cost optimum and keeps the neighbor-to-neighbor trust signal. **Recommendation: skip, as you suspected.** If a mandal ever wants bulk/automated sending, add the WhatsApp Business API later as an opt-in paid tier from the mandal's number.

### "Is the transparency link fully public?"

Yes, currently: once the admin publishes, anyone with `/transparency/<slug>` sees the aggregates (total collected + category pie — never donor names/amounts/phones). Your permission idea is good and cheap to add — see Feature F5.

### "Receipt link shows a UUID — can it be a slug/receipt number?"

The unguessable token is deliberate (it's the only thing keeping receipt data private), but we can make the URL human-friendly without weakening it — see Feature F4.

## Part 3 — Feature plan

### F1. SMS + WhatsApp send choice (SMS primary) — *mostly built already*
`send.ts` already has `buildWhatsAppLink`/`sendReceiptWhatsApp`, and PendingSend/CollectionForm have both buttons. Remaining work:
- On submit, keep auto-SMS as the primary default, but show the post-submit success state with two buttons: **Send by SMS** (primary) / **Send on WhatsApp** (secondary) — matching what PendingSend already does.
- Remember the volunteer's last-used channel (localStorage) and auto-open that one next time; keep the other as the visible alternative.
- Disable both when donor has no phone (new-issue #4).
- Track which channel was used if you ever want stats (optional column, skip for now).

### F2. Fix donor 404 — `vercel.json` rewrite (see above). **Do first.**

### F3. Receipt design upgrades (your specific notes + reference)
- Signature block: larger signature image; show **president name + "President" label** below it (currently label only).
- Header: larger mandal logo above the mandal name.
- Footer: inquiry contacts block (see F6).
- Preview: **"Preview donor receipt" button in Settings** rendering the real `ReceiptPage` with a sample donation and current branding — so the admin sees exactly what donors see after changing logo/signature.

### F4. Human-friendly receipt URLs
Change `/r/<128-bit-token>` → **`/r/<receipt_no>-<12-char token>`** (e.g. `/r/123-x7k9q2mfp3ab`), keeping ≥64 bits of entropy in the short token so links stay unguessable. Receipt number becomes visible in the link (nice for donors, harmless — the token still gates access). Server: lookup by token suffix only; `receipt_no` prefix is cosmetic and verified against the row. Old `/r/<full-token>` links keep working (redirect). Optionally prefix mandal slug (`/r/vinayak-mandal-123-…`) — decide by taste; longer links wrap worse in SMS, so I'd keep it short.

### F5. Transparency visibility control + block
Add `mandals.transparency_visibility`: `'public' | 'members' | 'admins' | 'disabled'` (default `public` to preserve current behavior).
- `public` — anyone with the link (today's behavior).
- `members` — must be signed in as a volunteer/admin **of that mandal**.
- `admins` — admins of that mandal only.
- `disabled` — link returns a friendly "report not available" (your "block" option).
Enforced in the `get_transparency_report*` RPCs (not client-side), radio picker in Settings, badge on the admin transparency screen showing current visibility. Publishing stays a separate toggle.

### F6. Inquiry contacts on receipt footer
- Settings: up to **2 extra contacts** (name + phone), president is the default contact.
- Toggle to hide the president's number — but if no other contact exists, president shows anyway (your rule).
- Receipt footer renders "For any questions: <name> — <number>" lines.
- Schema: `jsonb` column on `mandals` (`[{name, phone}]`) + `hide_president_contact boolean`; exposed through the public receipt RPC. Note: these numbers become public to anyone with a receipt link — that's the point, but the Settings copy should say so.

### F7. City + state in signup
Replace the bare state dropdown with a **city typeahead over a bundled JSON of Indian cities** (~1,500 entries, a few KB gzipped, zero API cost, works offline): type "Vadodara" → pick "Vadodara, Gujarat" → fills both `city` and `state`. Free-text fallback ("My city isn't listed") keeps nobody blocked. Schema: add `city` to `mandals`; keep `state`.

### F8. Navigation/back-button pass
`AppShell` already supports a `back` prop — it's just not applied everywhere. Sweep every screen:
- `/collect` (CollectionForm) — needs back-to-dashboard for admins / clear home for volunteers.
- Landing page sample report — needs an in-page back/close.
- Audit remaining screens (expenses, handover, cash-in-hand, transparency preview, receipt sent state) for a consistent top-left back.
- Rule: every screen reachable by tap must have an in-app back; browser/OS back should never be the only way out.

### F9. Admin dashboard redesign + overall UI pass — design reviewed ✅
Design reference captured in full via Chrome (all three views + transparency report); detailed extraction in `docs/design-reference-notes.md`. Key adoptions: dark-sidebar treasurer console with books-balance **equation** banner ("Net Balance ₹X = Volunteers ₹A + Treasurer cash ₹B + Bank ₹C"), styled stat trio with dark net-balance card, cash-in-hand tracker rows, volunteer bottom tab bar, quick-amount chips (₹101/251/501/1100/2100), maroon "you owe the treasurer" hero on the cash screen, and the full receipt composition (invocation line, formatted receipt no. VM/2026/0012, amount-in-words, name+title under a larger signature, scalloped edges). Principle held: our direct-send flow, WhatsApp option, offline tray, multi-tenant branding, and permission model stay; the reference contributes visual patterns only.
